# ISMS Documentation Development Workflow

1.	Need identified (e.g. based on ISO standard)
2.	Output defined (e.g. a policy) ideally lifting text directly from the ISO standard
3.	Ticket created in GitHub project by Product Owner – assign due/delivery date etc
4.	Ticket assigned to a developer
5.	Output created and uploaded to `SharePoint docs folder`
6.	SharePoint document linked to the GitHub ticket (as a comment)
7.	Output reviewed, edited, and approved (by “OMG”)
8.	Approved/final output turned into markdown and a new file is created in GitHub
9.	Ticket closed on GitHub 

## Notes

- We will need a more formal change management process as we approach certification. At this stage documents will change very infrequently.
- Once we have a few people working on tickets, start a daily/regular standup
